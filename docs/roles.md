# 角色、需求与交互模型

系统有两大业务域：**门诊运营管理系统**（流程）与**患者数字孪生体系统**（数据聚合）。下面梳理各角色如何与两者互动、需求如何被满足。这是实现前的需求澄清，结论会驱动 RBAC 与数据可见性设计。

## 1. 角色 × 系统 映射

| 角色 | 门诊运营系统 | 患者数字孪生体系统 | 主要诉求 |
|------|--------------|--------------------|----------|
| 前台 | ★ 主操作用户 | 只读（基本信息） | 登记/预约/签到/收费快、少出错 |
| 医生 | 接诊触发、写回 | ★ 主消费者/生产者 | 历史一目了然、辅助决策、开方写病历 |
| 护士 | 分诊/执行医嘱/随访 | 只读 + 生命体征写入 | 医嘱清晰、随访任务、体征录入 |
| 实习医生 | 草稿（上级审阅） | 只读 + 草稿 | 学习、带教审阅、知识库辅助 |
| 保安 | 在场/流量（受限） | 不可访问（PII） | 秩序、特殊关照标记、告警 |
| 患者 | 自助预约/看自己 | 看自己（隐私受限） | 自助、透明、提醒 |
| 患者家属 | 代预约/陪诊 | 看授权范围 | 代理操作、授权委托 |

## 2. 各角色需求细化

### 门诊运营管理系统
- **前台**：新建/修改预约（冲突检查）、查看当日排班与队列、患者到达签到/分诊、挂号收费入口、患者基本信息维护。
- **保安**：查看在场患者流量与特殊关照名单（仅必要字段）、接收异常告警；**不接触诊疗数据**。
- **医生**：从运营系统领取候诊患者 → 进入接诊。
- **护士**：分诊分级、执行/核对医嘱、录入生命体征、管理随访任务。
- **患者/家属**：自助预约、取消、候诊进度查询。

### 患者数字孪生体系统
- **医生**：接诊时拉取孪生体汇总（诊断时间线、用药、辨证演变），写病历/问诊后**回写刷新**孪生体。
- **护士**：读取孪生体以执行护理；写入生命体征（可作为孪生体新输入）。
- **实习医生**：只读或提交草稿，需上级（医生）审阅后才成为正式 records/consultations。
- **患者/家属**：查看本人/被授权范围的孪生体摘要；敏感字段（如特定诊断）可设可见性开关。

## 3. 典型交互流程（一次就诊闭环）

```
前台登记/预约 ──► 运营系统 appointments(+schedule)
        │
        ▼ 患者签到
运营系统 visits(checkin) ──► 进入候诊队列
        │
        ▼ 医生接诊
医生读取 患者数字孪生体(records+consultations 汇总)
        │  写 病历(records) / 问诊(consultations)
        ▼
孪生体系统 重新汇总(build_summary) ──► 刷新 twin 文档(版本化)
        │
        ▼ 护士
执行医嘱 / 录入生命体征 ──► 回写孪生体
        │
        ▼ 离院
家属(经授权)查看 孪生体摘要 / 候诊已结束
```

要点：**孪生体是派生视图**，由 records + consultations（+ 生命体征）汇总而成；任何临床写入都会触发其刷新（复用现有版本化与 FTS）。

## 4. 权限模型（RBAC，设计要点，待实现）

- 角色 → 权限集合（读/写/草稿/管理）。建议最小集合：
  - `reception`：运营 增删改查；孪生体 只读基础信息。
  - `doctor`：运营 接诊；孪生体 读写 + 刷新。
  - `nurse`：运营 执行；孪生体 读 + 体征写。
  - `intern`：运营 草稿；孪生体 只读 + 草稿（需 doctor 审阅）。
  - `security`：运营 在场/流量（受限字段）；孪生体 无。
  - `patient`：运营 本人预约；孪生体 本人（隐私受限）。
  - `family`：运营 授权内代操作；孪生体 授权内（隐私受限）。
- 认证只在 **API 层**加（见 architecture §安全），核心库不感知角色；权限在 API/服务层裁决。
- 当前 `NoOpProvider` 阶段**不实现 RBAC**，但表与接口要预留 `role`/`actor_id` 关联位。

## 5. 隐私与授权

- PII（病历/孪生体）加密落盘 + 访问审计（architecture §5）。
- 患者/家属仅可见本人或被显式授权范围；敏感诊断可标记可见性。
- 家属代理需"授权委托"记录（可在 `ops` 或独立 `consents` 表体现）。

## 6. 待确认决策（实现前需拍板）

1. **患者/家属是否提供自助端（App/小程序）？** 还是本期仅内部角色（前台/医护）使用，患者侧后续再做？
2. **RBAC 何时落地？** 与真实 LLM 同期，还是更早（运营/孪生体上线即需权限）？
3. **实习医生草稿审阅流**是否纳入本期，还是先当作普通只读？
4. **保安的"在场/流量"**数据来源（闸机/手动签到），是否本期就需要？

> 以上决定会影响 schema（如 `consents` 表、权限表）与 API 路由设计，建议先定再写代码。
