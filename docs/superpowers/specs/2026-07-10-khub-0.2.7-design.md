# khub 0.2.7 设计说明 — 临床孪生增强（四线并行）

- 日期：2026-07-10
- 状态：已批准（方案 A）
- 关联发布：v0.2.7
- 关联文档：`docs/architecture.md`、`docs/config.md`、`khub/clinical/*`、`khub/llm/*`、`khub/ops/*`

## 1. 目标与范围

在已发布的 v0.2.6（Diff 对比、老问诊导入器、defusedxml 安全加固、bandit/mypy/CI）基础上，聚焦**临床孪生子系统增强**，一次性合入四条相互独立的能力线：

1. 孪生摘要增强（增量 / 时间线 / 辨证脉络）
2. 问诊助手（对话式，基于孪生 + 知识检索）
3. 随访与复诊管理（库内状态标记，不接外部通知）
4. 病历 / 问诊结构化抽取（LLM + 离线词典兜底）

四条线均遵循「业务模块只加表、不改核心表」与既有横切约定（见 §2）。

### 不在范围内（Out of Scope，留待后续迭代）

- 体质画像 / 长期健康趋势建模
- 外部通知渠道（短信 / 微信 / 邮件）——MVP 仅库内状态标记
- 多用户 / 角色权限体系
- 知识检索（Retriever）本身的能力升级（仅复用，不改）

## 2. 横切约定（四条线强制共用）

1. **PII 安全**：新增表凡涉及患者内容的字段均经 `crypto.enc` / `crypto.dec`（仿 `khub/clinical/records.py` 的 `_PII_FIELDS`）；每次读取 PII 经 `audit.record_access` 留痕。
2. **离线兜底**：所有 LLM 调用统一走 `llm.get_provider()`；`NoOpProvider.complete` 返回空时回退规则 / 模板（仿 `khub/clinical/twin.py:_fallback_text`），**不得因无模型而报错或 500**。
3. **WAL 复制**：任意新增业务表必须 `install_triggers(conn, table, pk)` 接入复制日志，否则灾备 / 热备副本丢失该表数据。
4. **测试范式**：`Store(":memory:")` 内存库 + `monkeypatch.setenv`（`KHUB_LLM_URL` / `KHUB_PII_ENCRYPT`）或 `FakeProvider`；新增核心用例归入 `smoke` marker；真实 LLM / Embedding 路径打 `net` marker。
5. **传输层解耦**：CLI（`cli.py`）与 REST（`api.py`）均为薄消费者，业务逻辑只落在 `khub/clinical/*` 新模块内。

## 3. 能力线设计

### 3.1 孪生摘要增强

**改动点**：增强 `khub/clinical/twin.py`，新增游标表。

**新增表**（业务独立，接 WAL）：
```sql
twin_versions(
  id INTEGER PRIMARY KEY,
  patient_id INTEGER,
  base_record_id INTEGER,
  base_consult_id INTEGER,
  summary TEXT,            -- 经 crypto.enc 落盘
  created_at TEXT
)
```

**函数**：
- `build_summary_incremental(store, pid)`：读取 `twin_versions` 中该 `pid` 的最大 `(base_record_id, base_consult_id)` 游标；仅聚合 `records`/`consultations` 中 `id` 大于游标的新条目；prompt 附「在既有摘要基础上补充」，调 `get_provider().complete`；无新增条目则跳过并提示。保留现有全量 `build_summary` 作为兜底 / 重建入口。
- `get_timeline(store, pid)`：按 `records.visit_date` 与 `consultations.date` 排序，输出就诊时间线。
- `get_syndrome_evolution(store, pid)`：从历次 `differentiation` / `tongue_pulse` 抽取辨证脉络（复用 `Retriever` 做相似辨证检索，仅做横向对比，不做体质画像）。

**端点 / CLI**：
- `GET /twin/<pid>`（现有扩展，返回摘要 + 时间线 + 辨证脉络）
- `khub twin refresh <pid>`（`--full` 切换全量重建）

**测试**：无数据 / 有增量 / 无增量 / `NoOpProvider` 兜底 / PII 加密 + 审计；`twin_versions` 接 WAL 后 `replay_from` 可回放。

### 3.2 问诊助手

**改动点**：新增模块 `khub/clinical/consult_chat.py`。

**新增表**（接 WAL）：
```sql
consult_sessions(id INTEGER PRIMARY KEY, patient_id INTEGER, created_at TEXT)
consult_messages(
  id INTEGER PRIMARY KEY, session_id INTEGER, role TEXT,
  content TEXT,            -- 经 crypto.enc 落盘
  created_at TEXT
)
```

**函数**：
- `start_session(store, pid) -> session_id`
- `chat(store, session_id, user_msg) -> reply`：组装单 prompt =（孪生摘要 `build_summary` + `Retriever` 命中的知识片段 + 多轮 `consult_messages` 历史），调 `get_provider().complete(prompt)`；历史与摘要按 `khub/rag.py` 的 6000 字上限截断防超上下文；`NoOpProvider` 返回占位模板。写入 `consult_messages`（user + assistant 两条，均加密）。
- 会话内容读取经 `audit.record_access`。

**端点 / CLI**：
- `POST /clinical/consult/chat`（`{patient_id, session_id?, message}`）
- `khub consult-chat <pid>`（交互 / 单次 `--message`）

**测试**：首轮 / 多轮历史拼接 / `NoOpProvider` 占位 / 长上下文截断 / PII 加密 + 审计；`net` marker 覆盖真实模型。

### 3.3 随访与复诊管理

**改动点**：新增模块 `khub/clinical/followup.py`。

**新增表**（前缀 `followup_` 避命名冲突，均接 WAL）：
```sql
followup_plans(id INTEGER PRIMARY KEY, patient_id INTEGER,
  due_date TEXT, reason TEXT, status TEXT, created_at TEXT)
followup_reminders(id INTEGER PRIMARY KEY, plan_id INTEGER,
  due_date TEXT, channel TEXT, status TEXT, created_at TEXT)
followup_adherence(id INTEGER PRIMARY KEY, plan_id INTEGER,
  attended INTEGER, note TEXT, created_at TEXT)
```

**函数**：
- `add_plan(store, pid, due_date, reason) -> plan_id`（CLI 手工建）
- `scan_due(store, as_of=None)`：扫描 `due_date <= as_of` 且 `status != done` 的计划，库内标记 `status='due'`；可选调用 `ops.book_appointment` 自动落 `appointments`（仅当显式开启 `--auto-book`，默认关）。
- `record_adherence(store, plan_id, attended, note)`：写 `followup_adherence`。
- **MVP 仅库内状态标记，不接外部通知渠道**（短信 / 微信 / 邮件均留待后续）。

**调度接入**：`khub followup-scan` 作为 YAML 任务接入现有 `scheduler.run_tasks`（复用以 `shell=False` 执行命令的约定，禁用任意 shell）。

**端点 / CLI**：
- `khub followup-add <pid> --due <date> --reason`
- `khub followup-scan [--config <yaml>] [--auto-book]`
- `khub followup-adherence <plan_id> --attended/--missed`
- REST `/clinical/followup` 系列（list / add / adherence）

**测试**：建计划 / 扫描标记 due / 自动预约开关 / 依从性记录 / PII 加密；`net` 不依赖。

### 3.4 病历结构化抽取

**改动点**：新增模块 `khub/clinical/extract.py`。

**新增表**（接 WAL）：
```sql
record_struct(
  id INTEGER PRIMARY KEY, source TEXT,   -- 'record' | 'consult'
  source_id INTEGER,
  differentiation_norm TEXT, syndrome TEXT, formula TEXT, method TEXT,
  created_at TEXT
)
syndrome_vocab(canonical TEXT PRIMARY KEY, aliases TEXT)  -- JSON 数组，归一种子
```

**函数**：
- `extract_structured(store, text, provider) -> dict`：设 `KHUB_LLM_URL` 时 LLM 返回 JSON（辨证 / 证型 / 方剂 / 治法）；`NoOpProvider` 或异常时回退词典 / 正则匹配（复用 `khub/importer.py` 的 `_CONSULT_FIELDS` 作种子词表，经 `syndrome_vocab` 归一）。
- `apply_struct(store, source, source_id, structured)`：写 `record_struct`（`source` 区分 `record` / `consult`）；原始 `records` / `consultations` 列保持自由文本（核心表稳定），归一结果落新表。
- 字段归一化冲突（同一证型多写法）由 `syndrome_vocab` 仲裁。

**端点 / CLI**：
- `khub record-extract <record_id>` / `khub consult-extract <consult_id>`
- `POST /clinical/extract`（`{source, source_id, text?}`）

**测试**：LLM 命中路径（`net` marker）+ `NoOpProvider` 词典兜底路径（smoke）；PII 加密 + 审计；`syndrome_vocab` 归一仲裁。

## 4. 测试策略

- 每条线独立 `tests/test_clinical_twin_v2.py` / `test_consult_chat.py` / `test_followup.py` / `test_extract.py`（命名可调整）。
- 全部 `Store(":memory:")`，覆盖：无数据、正常路径、离线兜底、PII 加密 + 审计、WAL 可回放（`replay_from`）。
- 真实 LLM / Embedding 路径统一 `net` marker，离线 CI 跳过。
- 现有 283 用例（279 passed / 4 skipped）不得回退；0.2.7 目标用例数适度增长。
- 全量 `pytest -m "not slow and not net"` 须在 CI 保持绿。

## 5. 安全与风险

| 风险 | 缓解 |
|------|------|
| PII 泄露 | 新增表涉及患者字段均 `crypto.enc` + `audit.record_access`；REST 继承 `KHUB_API_TOKEN` 鉴权 |
| 离线无模型报错 | 所有 LLM 调用经 `NoOpProvider` 兜底，返回占位 / 模板，不抛 500 |
| 超上下文 | 问诊助手历史 + 摘要按 6000 字上限截断（仿 `rag.py`） |
| 灾备丢数据 | 新增表全部 `install_triggers` 接 WAL |
| 调度命令注入 | 复用 `scheduler` 的 `shell=False` 约定，禁用任意 shell |
| 抽取准确性 / 幻觉 | LLM 路径仅建议，最终以 `syndrome_vocab` 归一仲裁；离线兜底覆盖有限需在文档说明 |

## 6. 发布流程（沿用 0.2.6）

1. 四条线并行开发（4 个子 agent，各自功能分支，独立模块 / 表 / 测试）。
2. 统一合并至 `master`，全量测试绿。
3. `khub/__init__.py` 与 `pyproject.toml` 版本 bump `0.2.6 → 0.2.7`。
4. `CHANGELOG.md` 新增 `[0.2.7]` 小节（四线分类同本文）。
5. 提交、打 `v0.2.7` tag、推送、`gh release` 说明。

## 7. 验收标准

- 四条线均可独立运行；离线（无 `KHUB_LLM_URL`）下全部功能不报错、有合理兜底输出。
- 新增表均在灾备副本可回放（`replay_from` 验证）。
- 全量测试 `279+` passed、无回退；CI（bandit + smoke）绿。
- 版本号、CHANGELOG、tag、Release 说明齐全。
