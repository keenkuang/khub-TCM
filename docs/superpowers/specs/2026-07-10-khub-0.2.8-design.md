# khub 0.2.8 设计说明 — 运营界面与可观测性

- 日期：2026-07-10
- 状态：已批准
- 关联发布：v0.2.8
- 关联文档：`docs/architecture.md`、`khub/web/*`、`khub/ops/*`、`khub/log.py`、`docs/deployment.md`

## 1. 目标与范围

在已发布的 v0.2.7（临床孪生四线增强）基础上，分两条独立线推进：

### 线 A：Web UI + 运营子系统增强（B + D 合一）
聚焦前端体验与后台运营能力闭环，一次性合入：
- **Web UI 体验升级**（方向 B）：修复 PUT 编辑 format 降级 bug、运营 Web UI、搜索加载状态管理、移动端适配打磨
- **运营子系统增强**（方向 D）：预约/排班状态机完工、冲突检测、CLI 补齐、运营统计

### 线 B：运维可观测性（方向 F）
聚焦运维基础设施，独立于业务逻辑：
- 结构化日志 + 日志轮转
- `/health` 深度增强
- `/stats` 扩展
- `/metrics` Prometheus 端点
- 恢复 HTTP 访问日志

### 不在范围内

- 排班/预约的 WebSocket 实时推送
- 外部通知渠道（短信/微信/邮件）
- oAuth/SSO 鉴权
- OpenTelemetry 全链路追踪

## 2. 线 A：Web UI + 运营子系统增强

### 2.1 修复 PUT 编辑 format 降级 bug

**问题**：`PUT /documents/{id}` 未携带 `format` 字段（review_webui_r3.md NEW-4），HTML 文档编辑后降级为 plain。

**解决**：`api.py` 的 PUT 分支读取 `body.get("format")`，如缺失则从当前文档版本查询 `format` 字段补全，确保保存后格式不变。

**改动**：修改 `khub/api.py` dispatch PUT 分支（~10 行）

### 2.2 运营 Web UI

**现状**：前端零运营界面。`khub/web/` 只有文档/搜索/冲突/统计页面。

**新增**：
- `index.html` 导航栏新增「运营」入口
- `script.js` 新增 `loadSchedules()`、`loadAppointments()`、`loadVisits()` 函数
- 运营页面：排班列表（日期/医生/时段）、预约列表（患者/时间/状态/操作）、签到/取消按钮
- 复用现有样式与骨架屏模式

**改动**：修改 `khub/web/index.html`、`khub/web/script.js`、`khub/web/style.css`

### 2.3 搜索体验增强

- `script.js` 中为 `search()`/`semantic()` 添加 `AbortController`，新请求自动取消旧请求，消除结果竞争
- 添加网络错误提示 toast（fetch 失败时显示）
- 键盘快捷键：`/` 聚焦搜索框、`Escape` 清空/关闭详情

### 2.4 加载状态与移动端打磨

- 所有视图覆盖骨架屏（当前 stats/conflicts/sync-status 缺失）
- toast 消息持久化至操作反馈
- AI 面板在移动端 `@media` 中全屏显示

### 2.5 运营子系统增强

**状态机补全**：
- `appointments.status` 扩展：`booked` → `checked_in` → `completed`；新增 `cancelled`、`no_show`
- 新增 `cancel_appointment(store, id)`：设置 `status='cancelled'`
- 新增 `mark_no_show(store, appointment_id)`：设置 `status='no_show'`
- 新增 `complete_visit(store, visit_id)`：设置对应 `appointment.status='completed'`
- 新增 `reschedule_appointment(store, id, new_date)`：克隆原预约中 `patient_id/doctor`，设置原单 `cancelled`

**排班冲突检测**：
- `add_schedule(store, date, doctor, slot)` 检查同一 `(date, doctor, slot)` 是否已存在，存在则抛出 `ValueError`
- `book_appointment(store, patient_id, date, doctor)` 检查该 `(date, doctor)` 是否有 `stop`，且排班 slot 数未耗尽

**CLI 补齐**：
```bash
khub ops-schedule <date> <doctor> <slot>        # 新
khub ops-list [--date] [--doctor] [--status]    # 新
khub ops-cancel <appointment_id>                # 新
khub ops-reschedule <appointment_id> <new_date> # 新
khub ops-checkin <patient_id> <date> <doctor>   # 已有，保持
```

**运营统计（扩展 `/stats`）**：
在现有 `/stats` 返回中新增：
```json
{
  "appointments_today": N,
  "appointments_week": N,
  "appointments_by_status": {"booked": N, "checked_in": N, "completed": N, "cancelled": N, "no_show": N},
  "schedules_coverage": {"total_slots": N, "booked_slots": N, "utilization": 0.XX}
}
```

## 3. 线 B：运维可观测性

### 3.1 结构化日志

**改动**：重写 `khub/log.py`
- 默认日志格式从纯文本改为 JSON/NDJSON（每行一个 JSON 对象）
- 字段：`timestamp`、`level`、`module`、`message`、`request_id`（若在请求上下文内）、`duration_ms`
- 改用 `TimedRotatingFileHandler`（按天轮转，保留 `KHUB_LOG_ROTATION` 天，默认 30）
- 环境变量 `KHUB_LOG_FORMAT=text` 回退纯文本（兼容本地开发）
- 恢复 HTTP 访问日志：`api.py` 中 `self.server.RequestHandlerClass.log_message` 不再设为 `pass`，改为写入结构化日志

### 3.2 `/health` 深度增强

**扩展返回**：
```json
{
  "status": "ok" | "degraded",
  "version": "0.2.8",
  "uptime_sec": N,
  "checks": {
    "db": {"ok": true, "documents": N},
    "fts": {"ok": true, "fts_available": true},
    "disk": {"ok": true, "free_mb": N, "path": "~/.khub"},
    "llm": {"ok": true, "configured": false},
    "embedder": {"ok": true, "configured": false},
    "ha_role": "primary" | "standby",
    "wal_pending": N
  }
}
```
各检查项独立报告，单项失败返回 `"degraded"` 而非直接 500。

### 3.3 `/stats` 扩展

**新增字段**（保留现有文档维度字段）：
```json
{
  "...existing_fields": "...",
  "db_file_size_mb": N,
  "wal_pending_count": N,
  "requests_total": N,
  "errors_total": N,
  "requests_by_path": {"/search": N, "/stats": N, ...},
  "table_rows": {"twin_versions": N, "consult_messages": N, "followup_plans": N, ...},
  "llm_calls_total": N
}
```
请求计数通过 `api.py` 的 `dispatch` 入口处原子计数器实现（`threading.RLock` 保护）。

### 3.4 `/metrics` Prometheus 端点

- 无外部依赖，纯手写 Prometheus 文本格式输出（`# HELP` / `# TYPE` / 指标行）
- 采集指标：`khub_requests_total{path,method,status}`、`khub_request_duration_seconds{path}`、`khub_db_documents`、`khub_db_size_bytes`、`khub_wal_pending`、`khub_llm_calls_total`
- 默认**不启用**（设 `KHUB_METRICS_ENABLED=1` 显式开启），避免不必要的竞态

## 4. 横切约定

- **PII 安全**：运营子系统不新增 PII 字段，已有的可直接复用 `crypto.enc`；可观测性不涉及 PII
- **WAL 复制**：ops 现有三表均已 `install_triggers`，本次不新增表
- **测试**：`Store(":memory:")` + `smoke` marker；Web UI 功能走后端 API 测试（不测试前端 DOM）

## 5. 发布流程

1. 线 A（Web UI + ops）与线 B（可观测性）并行（2 个子 agent）
2. 统一合并至 `master`，全量测试绿
3. 版本 bump `0.2.7 → 0.2.8`、CHANGELOG、tag、gh release
