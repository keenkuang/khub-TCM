# khub M1 — 风险清单

- **日期**: 2026-07-09
- **来源**: 多角色评审（架构师/SWENG/安全/运维/PM）

---

## RISK-001: SQLite FTS5 不可用

| 字段 | 内容 |
|------|------|
| **描述** | 系统 Python 的 sqlite3 编译时未启用 FTS5 扩展，导致建表和检索均失败 |
| **概率** | 低 (Linux 发行版默认均启用；Windows/macOS 偶见缺失) |
| **影响** | 高 (M1 核心功能全文检索完全不可用) |
| **缓解措施** | 在 `Store.__init__()` 中运行时检查 FTS5 可用性并给出明确的安装指引错误消息；在 `conftest.py` 增加启动检查 |
| **应急措施** | 回退到 `LIKE '%keyword%'` 查询（无高亮、无排名），FTS5 推迟到解决系统依赖后 |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-002: 数据库文件损坏

| 字段 | 内容 |
|------|------|
| **描述** | SQLite 单文件在意外断电、进程崩溃、磁盘满等情况下可能损坏 |
| **概率** | 中 (个人工具使用场景以月/年计) |
| **影响** | 高 (全部文档和历史版本丢失) |
| **缓解措施** | Store 启用 WAL 模式降低损坏概率；提供 `PRAGMA integrity_check` 命令（M2）；文档说明定期备份 `khub.db` |
| **应急措施** | 从最新备份恢复；SQLite 提供 `.recover` 命令可尝试恢复部分数据 |
| **责任人** | 实施者(文档) / 用户(备份) |
| **状态** | 待缓解 |

## RISK-003: CLI 测试与实现框架不匹配

| 字段 | 内容 |
|------|------|
| **描述** | `tests/test_cli.py` 使用 `click.testing.CliRunner`，但实现用 `argparse`，测试无法编译 |
| **概率** | 高 (计划中同时包含了两种模式的测试代码) |
| **影响** | 中 (浪费 TDD 红-绿循环，CLI 功能不可验证) |
| **缓解措施** | 在进入实现前统一为 subprocess 测试模式，移除所有 click 引用 |
| **应急措施** | N/A (预防即可) |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-004: 中文 FTS5 检索零结果

| 字段 | 内容 |
|------|------|
| **描述** | FTS5 默认 `unicode61` tokenizer 按空白分词，不识别中文，"伤寒论"三字会被作为整体而非可拆分的 token |
| **概率** | 高 (若不主动处理，100% 发生) |
| **影响** | 高 (核心功能"检索"不可用) |
| **缓解措施** | 使用 `tokenize='trigram'` 替代默认 tokenizer；在 `test_db.py` 中增加中文检索验证测试 |
| **应急措施** | 回退到 `LIKE '%keyword%'` 查询 |
| **责任人** | 实施者 |
| **状态** | 已通过 ADR-006 缓解 |

## RISK-005: 集成测试意外写入生产数据库

| 字段 | 内容 |
|------|------|
| **描述** | 集成烟雾测试的默认 DB 路径硬编码为 `/home/keen/khub/khub.db`，运行测试可能覆盖真实数据 |
| **概率** | 中 (若测试在真实项目目录下运行) |
| **影响** | 高 (生产数据库被测试数据污染或覆盖) |
| **缓解措施** | 集成测试使用 `tmp_path` fixture 创建隔离的临时 DB 文件；删除 `KHUB_CONFIG=""` 这种触发默认路径的做法 |
| **应急措施** | 从备份恢复（若无备份则数据丢失） |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-006: 附件哈希全量读入内存

| 字段 | 内容 |
|------|------|
| **描述** | `OcrAdapter.pull()` 对附件文件调用 `f.read_bytes()` 计算 SHA256，大文件可能导致内存压力 |
| **概率** | 中 (OCR 扫描图单文件可达 10-50MB) |
| **影响** | 低 (个人使用，一次入库数本书，内存压力有限) |
| **缓解措施** | M1 文档中说明大附件场景；M2 使用流式哈希 |
| **应急措施** | 跳过附件哈希或将附件路径直接存储（不计算哈希） |
| **责任人** | 实施者(文档) |
| **状态** | 已接受 |

## RISK-007: Schema 变更导致数据迁移困难

| 字段 | 内容 |
|------|------|
| **描述** | M2+ 新增字段或表时，已有数据库无法自动获得变更 |
| **概率** | 高 (M2-M5 一定有 schema 变更) |
| **影响** | 中 (已有数据需迁移；`CREATE TABLE IF NOT EXISTS` 无法修改已有表) |
| **缓解措施** | 在 `Store.__init__()` 中实现 `schema_version` 检查 + 递增迁移脚本（M1 即预留机制） |
| **应急措施** | 手动导出数据 → 删除旧库 → 重新导入 |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-008: 依赖版本兼容性（PyYAML + Python 3.13）

| 字段 | 内容 |
|------|------|
| **描述** | PyYAML 6.0.x 对 Python 3.13 在某些平台（如 Linux arm64）没有预编译 wheel，需要 C 编译器 |
| **概率** | 低 (x86_64 Linux 有 wheel) |
| **影响** | 中 (pip install 失败或需要编译环境) |
| **缓解措施** | 在 `pyproject.toml` 中指定 `PyYAML>=6.0,<7`；在文档中说明若需要编译则安装 gcc/python3-dev |
| **应急措施** | 使用 `ruamel.yaml` 替代（纯 Python，无 C 扩展依赖） |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-009: 中文古籍内容编码问题

| 字段 | 内容 |
|------|------|
| **描述** | OCR 系统产出文件可能包含 Unicode 扩展区字符（生僻汉字、异体字、古籍用字）或非 UTF-8 编码 |
| **概率** | 中 (中医古籍数字化常见) |
| **影响** | 中 (内容显示错误、FTS5 检索不到、排序异常) |
| **缓解措施** | 在 OcrAdapter 中读取文件时使用 `encoding="utf-8"` 并捕获 `UnicodeDecodeError`；在测试中使用含生僻字的测试用例 |
| **应急措施** | 通知用户转换文件编码 |
| **责任人** | 实施者 |
| **状态** | 待跟踪 |

## RISK-010: `khub sync` 占位导致用户困惑

| 字段 | 内容 |
|------|------|
| **描述** | 用户自然倾向于先运行 `khub sync`，但 M1 中该命令仅打印占位消息 |
| **概率** | 高 (CLI 第一个子命令) |
| **影响** | 低 (操作失败损失时间，不损失数据) |
| **缓解措施** | 改进提示语为 "M1 只有 OCR push-in 源，请用 `khub ingest --book <dir>` 代替 sync。其他源的 sync 在 M2 实现。" |
| **应急措施** | 将 sync 从 M1 CLI 中暂时移除，待 M2 实现后恢复 |
| **责任人** | 实施者 |
| **状态** | 待缓解 |

## RISK-011: multi-source 文档去重

| 字段 | 内容 |
|------|------|
| **描述** | 同一文档从不同源(OCR/Obsidian/ima)分别入库，导致重复，占用空间且检索结果冗余 |
| **概率** | 中 (M2 多源上线后出现) |
| **影响** | 中 (检索质量下降、存储浪费) |
| **缓解措施** | M1 即预留 `source_ids` 字段，`store_document` 中实现简单的源 ID 去重逻辑 |
| **应急措施** | 手动清理重复文档 (CLI 提供 `dedup` 子命令) |
| **责任人** | 实施者 |
| **状态** | 待跟踪 (M1 不实现，ADR-009 记录) |
